## 完善缓存管理功能

### 任务描述
用户要求完善缓存管理功能，用于优化重复查询（如天气信息）。具体需求：
- 如果用户上一次查询了"今天上海天气"，已经调用过查询天气工具
- 下一次再问"今天上海天气"时，不要再调用工具，直接从缓存中获取
- 需要注意时间、地点的一致性
- 如果下一次问"明天上海天气"，那就要继续调用查询天气工具

### 完成的工作

#### 1. 修改 `backend/tools/weather_tool.py`
- 导入 `CacheManager` 并初始化缓存管理器实例
- 在 `get_weather` 函数开头添加缓存检查逻辑
  - 将 `today`/`tomorrow` 转换为实际日期
  - 先检查缓存是否命中
  - 如果缓存命中，直接返回缓存数据（打印日志 `[缓存命中]`）
  - 如果缓存未命中，调用 API 并将结果存入缓存（TTL = 3600秒 = 1小时）
- 添加详细的日志输出，方便调试

#### 2. 修改 `backend/tools/sunset_tool.py`
- 导入 `CacheManager` 并初始化缓存管理器实例
- 在 `get_sunset` 函数开头添加缓存检查逻辑
  - 将 `today`/`tomorrow` 转换为实际日期
  - 先检查缓存是否命中
  - 如果缓存命中，直接返回缓存数据（打印日志 `[缓存命中]`）
  - 如果缓存未命中，调用 API 并将结果存入缓存（TTL = 86400秒 = 24小时）
- 添加详细的日志输出，方便调试

#### 3. 创建测试脚本 `backend/test_cache.py`
- 测试天气查询缓存功能

  - 不同日期查询（调用新API）
- 测试日落时间查询缓存功能

- 天气查询缓存：正常工作
- 日落时间查询缓存：正常工作

### 技术细节
- 缓存键格式：`weather:{city}:{date}` 或 `sunset:{city}:{date}`
- 日期处理：在函数开头将 `today`/`tomorrow` 转换为 `YYYY-MM-DD` 格式
- TTL 设置：
  - 天气缓存：1小时（3600秒）- 因为天气变化较快
  - 日落时间缓存：24小时（86400秒）- 因为日落时间每天变化很小
- 缓存后端：支持 Redis 和内存缓存（通过 `CacheManager` 自动选择）

- 日志格式：`%(asctime)s - %(name)s - %(levelname)s - %(message)s`
